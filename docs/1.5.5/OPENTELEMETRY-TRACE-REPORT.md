**Date:** 2026-06-27
**Scope:** AISF distributed tracing implementation

## Architecture

OpenTelemetry tracing is implemented for AISF with optional OTLP export. It is not enabled by default (no collector required to run the platform), but activates automatically when `OTEL_EXPORTER_OTLP_ENDPOINT` is set.

```
Desktop
  │  X-Correlation-ID header (UUID, echoed in response)
  │  X-Request-ID header (new UUID per request)
  ▼
AISF FastAPI
  ├─ FastAPIInstrumentor (auto-instruments all routes)
  ├─ CorrelationMiddleware (binds correlation_id, request_id to ContextVar)
  ├─ enrich_span() copies ContextVar → active OTEL span attributes
  │
  └─ BatchSpanProcessor
       │
       ├── [if OTEL_EXPORTER_OTLP_ENDPOINT set] → OTLP/HTTP → Jaeger/Tempo/OTEL Collector
       └── [if AISF_OTEL_CONSOLE=1]             → ConsoleSpanExporter (stdout JSON)
```

## Configuration

**Module:** `factory/tracing/setup.py`

```python
def configure_tracing(
    service_name: str = "ai-software-factory",
    service_version: str = "1.5.4",
) -> None:
    resource = Resource.create({
        "service.name": service_name,
        "service.version": service_version,
        "deployment.environment": os.environ.get("AISF_ENV", "local"),
    })
    provider = TracerProvider(resource=resource)
    otlp_endpoint = os.environ.get("OTEL_EXPORTER_OTLP_ENDPOINT", "")
    if otlp_endpoint:
        exporter = OTLPSpanExporter(endpoint=otlp_endpoint)
    elif os.environ.get("AISF_OTEL_CONSOLE"):
        exporter = ConsoleSpanExporter()
    else:
        exporter = None  # no-op spans — no overhead
    if exporter:
        provider.add_span_processor(BatchSpanProcessor(exporter))
    trace.set_tracer_provider(provider)
```

**Environment variables:**

| Variable | Effect |
|---|---|
| `OTEL_EXPORTER_OTLP_ENDPOINT` | Enable OTLP export (e.g., `http://localhost:4318`) |
| `AISF_OTEL_CONSOLE=1` | Enable console (stdout) span export for debugging |

## Correlation ID Propagation

The `X-Correlation-ID` header links a single user action across all three services:

1. **Desktop** generates or forwards `X-Correlation-ID` on every API call (via `ApiClient._timed_request`)
2. **AISF CorrelationMiddleware** reads the header (or generates new UUID), binds to ContextVar, echoes in response header
3. **AISF → CF calls** include the same `X-Correlation-ID` (when using AISF as proxy)
4. **CF CorrelationFilter** reads `X-Correlation-ID`, puts in MDC → appears in every structured log line as `correlation_id`

This creates a searchable chain: grep any log file for the correlation ID to see the complete request path across all services.

## Cross-Service Log Correlation Example

Desktop log (`runtime/logs/desktop.log`):
```json
{"ts":"2026-06-27T12:00:01.234Z","msg":"API call POST /api/v1/episodes","correlation_id":"a3b4c5d6","duration_ms":12.4}
```

AISF log (`logs/aisf.log`):
```json
{"ts":"2026-06-27T12:00:01.238Z","msg":"POST /episodes → 201","correlation_id":"a3b4c5d6","request_id":"7f8e9a0b","worker_id":"STORY_GEN"}
```

CF log (`logs/content-factory.log`):
```json
{"@timestamp":"2026-06-27T12:00:01.240Z","message":"StoryGenerationWorker processing","correlation_id":"a3b4c5d6","worker_id":"StoryGenerationWorker","episode_id":"01HXA3..."}
```

All three log lines share `correlation_id: a3b4c5d6` — full cross-service trace reconstructable from logs alone, even without a Jaeger/Tempo backend.

## Jaeger Integration (Optional)

To enable full distributed trace visualization:

```bash
# Start Jaeger all-in-one
docker run -d --name jaeger \
  -p 16686:16686 -p 4318:4318 \
  jaegertracing/all-in-one:latest

# Set env var before starting AISF
set OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4318

# Open Jaeger UI
start http://localhost:16686
```

Search for service `ai-software-factory`, filter by `aisf.correlation_id` tag.
