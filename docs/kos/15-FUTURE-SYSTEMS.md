# KOS Future Systems Dependency

## The Rule

> All future systems MUST depend on KOS. No exceptions.

This document specifies exactly what "depend on KOS" means for each future system,
and what the CI gate looks like for each.

---

## Dependency Pattern

Every future system follows this pattern:

```
1. Register all its components as Knowledge Objects in KOS
2. Declare all KOS dependencies in its build configuration
3. Pass platform-kos validate --system {system_id} before deployment
4. Emit runtime observations back to KOS
```

---

## System 1 — Platform Kernel

**Phase:** 3.0F (Platform Migration)  
**KOS Dependency Type:** Foundational  
**What it registers in KOS:**
- Kernel subsystems (DI, Events, Config, Lifecycle, Registry) as `KernelObject`
- Every kernel interface as `CapabilityObject`
- Boot sequence specification as `ArchitectureObject`

**CI Gate:**
```bash
platform-kos validate --system platform-kernel
# All kernel objects CANONICAL before merge
```

---

## System 2 — AI Runtime

**Phase:** 3.0G (AI Runtime Migration)  
**KOS Dependency Type:** Full (all 29 modules registered)  
**What it registers:**
- All 29 AI runtime modules as `ModuleObject`
- All 32 capabilities as `CapabilityObject`
- All 10 algorithms as `AlgorithmObject`
- All 160+ tests as `TestObject`

**CI Gate:**
```bash
platform-kos validate --system ai-runtime
pytest platform/tests/ --kos-coverage-min 90
```

## System 3 — Software Factory

**Phase:** 3.0H (Software Factory Migration)  
**KOS Dependency Type:** Full + Agent  
**What it registers:**
- All orchestration modules as `ModuleObject`
- All agent definitions as `AgentObject`
- All task types as `TaskObject`
- Dispatch strategies as `StrategyObject`

**KOS Consumed:**
- Execution plans generated by KOS Execution Engine
- Agent definitions compiled from KOS AgentObjects

**CI Gate:**
```bash
platform-kos validate --system software-factory
platform-kos validate --agents --min-quality 0.80
```

## System 6 — Research Runtime

**Phase:** Post-3.0I  
**KOS Dependency Type:** Research + Knowledge Producer  
**What it registers:**
- Research questions as `RequirementObject`
- Research plans as `ExecutionPlan` (type=research)
- Research findings as new `KnowledgeObject` instances

**Special:** Research Runtime is a **KOS producer** — it generates new knowledge objects
and submits them to the registry. It is the only runtime that adds to KOS autonomously.

## Universal CI Gate

For any new system added to AI Studio:

```bash
# Required before any system is deployed
platform-kos validate --system {system_id} --all

# Checks:
# - All components registered as knowledge objects
# - Minimum 1 requirement per module (GR-008)
# - Minimum quality score 0.70
# - Full traceability chain exists (GR-008)
# - Tests linked to knowledge objects (GR-006)
```

**A system that fails this gate may not be deployed.**
